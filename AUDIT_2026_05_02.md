## P0 — Critical (Branch 1: `fix/critical-bugs-batch`)

### From external audit
1. **Newsletter duplicated on homepage** — confirmed: rendered at [Home.tsx:155](client/src/pages/Home.tsx#L155) (card variant) AND inside Footer at [Footer.tsx:97](client/src/components/sections/Footer.tsx#L97) (inline variant). **Fix**: remove the inline version from Footer; keep dedicated section on Home. Footer keeps social links + CTA.

2. **5 distinct Launch App CTAs on homepage** — Navbar (desktop + mobile drawer), HeroSection MagneticButton, FloatingLaunchButton, Footer launch button, WelcomePopup launch link. **Fix**: hide Navbar Launch App button until user scrolls past hero (only shows when hero CTA is no longer visible). Hide FloatingLaunchButton on routes where Navbar's Launch App is showing. WelcomePopup is one-shot per session — leave it. Result: at most 2 visible at once.

3. **Loading flash on `HomeNumbersTeaser` stats** — confirmed: `CountUp` initializes `useState(0)`, so on first paint the section shows `0 / 0+ / 0 / $0K` for ~150ms before animation kicks in. **Fix**: render the final value as the initial state (skip animation if `prefers-reduced-motion`); animate only after `useInView` triggers. Audit also: `HeroSection` has the same CountUp pattern at L63-66.

4. **`/blog/:slug` renders 2 H1s** — confirmed: [BlogPost.tsx:243-248](client/src/pages/BlogPost.tsx#L243) renders the title as `<h1>`, AND [BlogContent.tsx](client/src/components/BlogContent.tsx) doesn't override the markdown `h1` mapping in its Streamdown components. If a post's markdown content starts with `# Title`, that becomes a 2nd `<h1>`. **Fix**: add an `h1` override in BlogContent.tsx that renders as `h2` (or strip leading `# Title` lines from posts at content-load time).

5. **Invalid blog slugs render fake "Post not found" inline** — confirmed: [BlogPost.tsx:142-148](client/src/pages/BlogPost.tsx#L142) renders inline message instead of NotFound component → URL stays, homepage SEO inherited. Same pattern in [TopicPage.tsx:214-230](client/src/pages/TopicPage.tsx#L214) and [ReelPage.tsx:322-330](client/src/pages/ReelPage.tsx#L322). FilmPlayer.tsx:14 falls through to FilmsPage when slug invalid. **Fix**: each of these renders `<NotFound />` (the existing 404 component) when its query/lookup returns null. Fixes SEO + UX in one shot.

6. **SubmitPage form labels not associated** — confirmed: 5 `<label>` tags exist (e.g., [SubmitPage.tsx:238](client/src/pages/SubmitPage.tsx#L238)) but none have `htmlFor`. Inputs lack `id`. Screen reader users can't tell which label belongs to which input. **Fix**: add unique `id` to each input/textarea/select; matching `htmlFor` on each label; `aria-required="true"` on required fields.

7. **Mobile video disappears/scrolls when played** — investigation findings:
   - `CinematicLightbox.tsx:227` has `playsInline` ✓
   - `ReelsSection.tsx:285` has `playsInline` ✓
   - Body scroll lock present in both ✓
   - `ReelPage.tsx:209-217`: `<video autoPlay playsInline loop muted={muted}>` — has `playsInline` ✓
   - **Likely cause** (without a real device, this is best-guess): the lightbox uses `min-h-screen`/`100vh` somewhere, and iOS Safari's dynamic URL bar resize causes layout shift. Modal animation may also be triggering scroll on mount.
   - **Proposed plan** (low-risk):
     - Replace `100vh` / `h-screen` with `100dvh` / `min-h-[100dvh]` (dynamic viewport height — adjusts for iOS toolbar) in lightbox/player containers.
     - Audit any `scrollIntoView` calls in video components — none found in the agent scan.
     - Ensure `<video>` `autoPlay` always pairs with `muted` initially (already correct).
     - Add `style={{ WebkitTransform: "translateZ(0)" }}` to the lightbox wrapper to force composite layer (fixes iOS rendering glitches on fixed positioning).
   - **Risk**: low — dvh has 95%+ browser support; transform hack is harmless on non-iOS.

8. **Missing alt text on 14 images** — `AdminDashboard.tsx:125` has `alt=""` on a content image (blog cover). Other "decorative" empty alts (flags, avatar rotators) are appropriate but the **community wall images at [SocialWallSection.tsx:130](client/src/components/sections/SocialWallSection.tsx#L130) ARE content** (real submitted photos), not decoration. **Fix**: descriptive alt for any image that conveys meaning; keep `alt=""` only for icons/decorations.

9. **SEO meta over Google's truncation limits** — homepage title 63 chars, description 192 chars. Films/Ecosystem descriptions ~190 chars. **Fix**: trim titles to ≤55 chars, descriptions to ≤155 chars. Pull most clickable words to the front. Touch points: [Home.tsx:42-43](client/src/pages/Home.tsx#L42), [FilmsPage.tsx:26](client/src/pages/FilmsPage.tsx#L26), [EcosystemPage.tsx](client/src/pages/EcosystemPage.tsx), [BlogPost.tsx:73](client/src/pages/BlogPost.tsx#L73).

10. **Share button AbortError → double-UI** — confirmed in BOTH share components:
    - [SharePagePill.tsx:46-57](client/src/components/SharePagePill.tsx#L46) — `nativeShare` returns `false` on cancel; caller opens custom popup at L77.
    - [ShareButton.tsx:95-98](client/src/components/ShareButton.tsx#L95) — `.catch(() => setOpen(true))` triggers modal on AbortError.
    - **Fix**: in both, distinguish AbortError (silent return) from real failure (custom fallback).

## Mobile video bug — fix plan (P0 #7)

**Investigation**: all video components have `playsInline` and `muted` correctly. Body scroll lock is in place. No `scrollIntoView` calls in video files. Modals use `fixed inset-0`.

**Most likely root cause**: iOS Safari's dynamic URL bar (visible/hidden depending on scroll position) changes the visual viewport height. Combined with `100vh` units and `position: fixed`, this can cause the lightbox container to jump.

**Plan** (low-risk, no behavior changes):
1. In `CinematicLightbox.tsx`, replace any `100vh` / `h-screen` with `100dvh` / `min-h-[100dvh]`. The `dvh` unit is dynamic — adjusts as iOS toolbar shows/hides.
2. In `ReelsSection.tsx` `ReelPlayer` overlay, same swap.
3. In `ReelPage.tsx` add `min-h-[100dvh]` to the wrapper.
4. Add `transform: translateZ(0)` (force compositor layer) to the lightbox root — fixes iOS Safari position-fixed glitches.
5. Ship + smoke-test routes. If a real device still reports the bug, iterate from there.

**Risk**: low. `dvh` has wide modern browser support; `translateZ(0)` is a long-known iOS workaround with no negative side effects.

**Approval**: proceeding autonomously per session protocol — change is targeted, well-understood, doesn't touch cron/auth.
